What Is a Pipeline?

A pipeline is the sequence your code follows before real users can use it. Think of it as a quality control system for software.

Workflow

Building Feature-by-Feature

The biggest mistake with AI tools is asking for too much at once. Professional developers build one isolated feature at a time, merging it safely before starting the next.

The AI Chaos Anti-Pattern

Feeding an AI a massive prompt to generate five features at once, directly on your main branch. The AI will overwrite shared files, break existing logic, and introduce bugs that won't surface until your users encounter them.

Step 1: Branch Isolation

Never code directly on dev, staging, or main. Ensure your local dev is up-to-date, then pull a fresh isolated branch for exactly one task.

bash 
git checkout dev
git pull origin dev
git checkout -b feature/user-login

Step 2: The Focused AI Prompt

With your isolated branch active, narrow your AI assistant's scope completely. Scope creep starts with undisciplined prompts.

Developer prompt to AI

I am working on the isolated feature/user-login branch. Implement only the login UI component and its authentication logic using Supabase. Do not touch any other features.

Step 3: The Pull Request

A PR is not a merge. It is a formal request: "I finished this feature. Review this isolated change and grant permission to merge." Use Conventional Commits so your git history is machine-readable and self-documenting.

bash 
git add .
git commit -m "feat(auth): add supabase email and password login"
git push origin feature/user-login
# Open PR on GitHub: feature/user-login → dev
# CI runs automatically when the PR is opened

Infrastructure

Environments & Branch Strategy

Isolated environments protect real users. A structured branch strategy controls how code flows between them.

DEV Experimental

Active development zone. Multiple features combined. Full admin access. Dummy data only. Never shown to users.

  • Safe to break
  • Shared by team
  • Frequent deploys
  • No real data
QA Testing

First serious checkpoint. All automated tests run here. QA engineers and developers test integration. Sanitized test data.

  • Full test suite runs
  • Regression testing
  • Manual QA testing
  • Security scanning
STAGING Pre-Production

Exact clone of production. Same database structure, same APIs, same server configs. Performance and stability testing.

  • Mirrors production exactly
  • Separate database
  • Performance testing
  • Real device testing
PROD Live

Real users. Real data. Only code that passed DEV, QA, and STAGING reaches here. Protected by every previous stage.

  • Protected by all stages
  • Real customer data
  • Automated deployments only
  • 24/7 monitoring

The Rules

  • Always branch from dev for new features
  • Open a PR to merge back to dev
  • Only merge dev → QA → staging → main
  • Never push directly to main
  • Never skip stages
  • Never code directly on staging

Branch Protection (GitHub Settings)

Go to: Repository → Settings → Branches → Branch Protection Rules → Protect main
  • Require pull request before merging
  • Require status checks to pass (CI must pass)
  • Require at least 1 code review approval
  • Block direct pushes (even from admins)
  • Dismiss stale reviews when new commits pushed
  • Require linear history (squash merge preferred)
Why squash merge? Squash merge combines all PR commits into ONE clean commit on main. This gives you a readable git history where each commit = one complete feature. If something breaks, git revert removes the entire feature cleanly.
CODEOWNERS: Create .github/CODEOWNERS to auto-assign reviewers based on file paths. Critical for AI-generated code: ensures the auth expert reviews auth changes, the DB team reviews migrations. GitHub enforces this via branch protection.
bash 
# .github/CODEOWNERS

# Database migrations require DBA team review
db/migrations/** @dba-team

# Auth changes need senior review
src/auth/** @alice @bob

# All JS requires any frontend dev
*.js @frontend-team
Production Safety

Shipping and Protecting Production

Deployment strategies, database migrations, secrets management, monitoring, and rollback: everything that keeps production healthy after code ships.

Deployment Strategies

How you push code to production determines your risk profile.

Blue-Green Deployment Zero downtime

Run two identical production environments. Blue is live; Green is the new version. When Green is tested, flip the load balancer. Instant rollback in under 1 second.

Best for: zero-downtime requirements, instant rollback capability.

Canary Deployment Risk-averse

Gradually shift traffic to the new version: 5% → 25% → 50% → 100%. Monitor error rates and latency at each step. If problems appear, rollback before all users are affected.

Best for: large user bases, high-traffic production systems.

Rolling Deployment Cost-efficient

Update servers one at a time. No extra infrastructure cost, but old and new versions run simultaneously during deployment. Rollback is slower than blue-green.

Best for: cost-sensitive projects with stateless applications.

Feature Flags: Deploy Without Releasing

Feature flags let you ship code to production while keeping it hidden. Flip a switch when you're ready. This decouples deployment from release.

  • Instant on/off without redeployment
  • Gradual rollout (show to 5% of users first)
  • A/B testing features
  • Kill switch if something goes wrong

Tools: LaunchDarkly, Unleash, Flipt

Database Migrations: The Danger Zone

Never make breaking database changes in a single step. Use the Expand/Contract pattern.

1

Expand

Add new column alongside old one. Both exist. Old code still works.

2

Migrate

Deploy new code that uses the new column. Old column still there as a safety net.

3

Contract

After confirming stability, remove the old column cleanly.

Changing a column name in one step while old servers are still running? That's how you get a 2am production outage.

Never Store Secrets in Code

typescript 
// NEVER DO THIS — this gets committed to git
const apiKey = "sk-1234567890abcdef";
const dbUrl = "postgres://user:password@host/db";
  • Store secrets in a vault (Doppler, HashiCorp Vault, AWS Secrets Manager)
  • CI/CD fetches secrets at runtime, never stored on disk
  • Different secrets per environment (staging DB ≠ production DB)
  • Auto-rotate credentials regularly
  • Commit .env to .gitignore, always

Monitoring

Sentry: Error Tracking

Catches when things break. Real-time alerts, stack traces, automatic grouping of similar errors. Get paged before users complain.

PostHog: Product Analytics

Measures if things matter. User behavior, feature adoption, A/B test results. Know what users actually do.

Rollback Strategy

When production breaks: can you fix it in under 5 minutes? Hotfix. Can't? Rollback immediately.
  • Vercel: Instant rollback; point domain to previous deployment in under 1 second
  • Railway: Git-based rollback; checkout previous commit and rebuild
  • Rule: Always keep the last 3 deployments available for instant rollback
Quality Assurance

Testing Strategy

Tests are the safety net that catches bugs before users do, and the primary guardrail that keeps AI-generated code safe.

Where Test Code Lives

Common confusion: the ci.yaml pipeline file does not contain your tests. It only runs the terminal command (npm test). Your actual test code lives inside your project, right next to the application files it tests.

Project structure
my-app/
├── .github/workflows/ci.yamlThe pipeline script — runs commands, not test code
├── src/
│ ├── components/
│ │ ├── Dashboard.tsxThe real app feature
│ │ └── Dashboard.test.tsx← Unit test: verifies this component in isolation
│ └── utils/
│ ├── discount.tsA utility function
│ └── discount.test.ts← Unit test: checks edge cases and math logic
└── package.jsonDefines the "npm test" command
No tests folder yet? Three steps: (1) Install a framework: npm install -D vitest. (2) Create a test file alongside any existing function. (3) Use AI for the heavy lifting: pass your service file to Claude and ask it to write unit tests covering edge cases — once the framework is wired up, this takes seconds.
E2E Tests (10%)
Playwright / Cypress: slowest, most realistic
Integration Tests (20%)
API + Database, service interactions
Unit Tests (70%)
Fastest, cheapest, most numerous
A bug caught by a unit test costs $1 to fix. The same bug in production costs $1,000+. Run thousands of unit tests, dozens of integration tests, and a handful of E2E tests.
Regression Testing: The Most Critical Test

Regression means: did the new feature accidentally break old features? This is especially dangerous with AI coding because agents don't fully understand your entire system. They may modify shared logic unexpectedly.

Example: You add workout streaks. Now login breaks, notifications stop working, profile images disappear. That's regression.

Rule: Run your FULL test suite after every feature merge. Automate it in CI so it can never be skipped.

Unit Tests

What Unit Tests Do

Test individual functions in isolation: no database, no network, no dependencies.

typescript 
describe('calculateDiscount', () => {
  it('applies 10% for orders over $100', () => {
    expect(calculateDiscount(150)).toBe(15);
  });
  it('applies no discount for exactly $100', () => {
    expect(calculateDiscount(100)).toBe(0);
  });
  it('throws for negative amounts', () => {
    expect(() => calculateDiscount(-50))
      .toThrow('Amount cannot be negative');
  });
});

What NOT to Test

  • Framework internals (React rendering, Express routing)
  • Implementation details (HOW it works)
  • External library behavior
  • Trivial getters with no logic

Integration Tests

Integration tests verify that multiple components work together: your API + database, your service + external APIs.

typescript 
describe('POST /users', () => {
  it('saves user to database and returns with ID', async () => {
    const response = await request(app)
      .post('/users')
      .send({ email: 'alice@example.com', name: 'Alice' });

    expect(response.status).toBe(201);
    expect(response.body.id).toBeDefined();

    // Verify it actually saved
    const saved = await db.query(
      'SELECT * FROM users WHERE email = $1',
      ['alice@example.com']
    );
    expect(saved.rows.length).toBe(1);
  });
});

E2E Tests

E2E tests simulate a real user in a real browser, walking through complete workflows.

typescript 
test('complete checkout flow', async ({ page }) => {
  await page.goto('https://shop.example.com');
  await page.fill('input[placeholder="Search"]', 'headphones');
  await page.click('text=Wireless Headphones');
  await page.click('button:has-text("Add to Cart")');
  await page.click('button:has-text("Checkout")');
  await page.fill('[name="cardNumber"]', '4111111111111111');
  await page.click('button:has-text("Complete Purchase")');
  await expect(page.locator('text=Order Confirmed')).toBeVisible();
});

Static Analysis

TypeScript catches entire categories of bugs before tests even run.

typescript 
// JavaScript — crashes at runtime
function greet(user) {
  return user.name.toUpperCase(); // What if user is null?
}

// TypeScript — error in your editor, instantly
function greet(user: { name: string }) {
  return user.name.toUpperCase(); // Safe — TypeScript guarantees it
}

greet(null); // ❌ TypeScript: Argument of type 'null' not assignable
Teams using TypeScript report 38% fewer production bugs (with fewer tests) because the compiler catches type errors before code runs.

Run SAST (Static Application Security Testing) on every PR.

Snyk

Catches SQL injection, XSS, hardcoded secrets, vulnerable dependencies. Free tier available.

CodeQL

GitHub's built-in scanner. Catches security vulnerabilities using query-based analysis. Free for public repos.

ESLint Security Plugins

Catches JavaScript-specific security issues directly in your editor and CI pipeline.

Code Coverage

Aim for 70–80% coverage on critical code. 100% is a trap: it incentivizes testing trivial getters instead of edge cases.
Coverage ROI
0–70% High ROI: each test catches real bugs
70–80% Strong ROI: edge cases covered
80–90% Moderate ROI
90–100% Diminishing returns: testing trivial getters
100% coverage can give false confidence. A test that executes code isn't the same as a test that verifies correct behavior.
Automation

CI/CD: The Automation Engine

Continuous Integration and Continuous Deployment automate the boring, critical work of testing and shipping code.

Continuous Integration (CI)

Merge code frequently, multiple times per day. Every merge triggers automated tests. Catch incompatibilities immediately rather than after weeks of divergence.

Small merges = small conflicts. The math is brutal in CI's favor.

Continuous Delivery vs Deployment

Delivery: Code is ALWAYS ready to deploy. Humans decide WHEN to press the button. Used in regulated industries.

Deployment: Code AUTOMATICALLY deploys the moment it passes all checks. Ideal for web apps and APIs.

What Happens When CI Fails

CI Fails
  • PR gets a prominent red ✗
  • Merge button stays locked
  • Exact error surfaced in logs
How to fix it:
Click "Details" on the failed check — it shows the exact file and line number causing the error (e.g., Type 'null' is not assignable in UserProfile.tsx:42)
Fix the issue on your local feature branch
Commit and push — CI reruns automatically on the fresh code
CI Passes
  • PR gets a green ✓
  • Merge button unlocks
  • Ready for human code review
  • Reviewer can approve and merge

GitHub Actions Anatomy

yaml 
name: CI Pipeline

on:
  push:
    branches: [dev]
  pull_request:
    branches: [dev, staging, main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm' # Caches node_modules between runs

      - name: Install dependencies
        run: npm ci # Faster than npm install, uses lockfile

      - name: Lint
        run: npm run lint

      - name: Type check
        run: npm run typecheck

      - name: Run tests
        run: npm test

      - name: Security scan
        run: npx snyk code test --severity-threshold=high

      - name: Build
        run: npm run build

  deploy-staging:
    needs: test # Only runs if test job passes
    if: github.ref == 'refs/heads/staging'
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Railway Staging
        run: railway up --environment staging
Dependency Caching: Without caching, 15-minute pipeline. With caching, 3-minute pipeline. cache: 'npm' in setup-node stores node_modules between runs. The cache key is the hash of your package-lock.json, so it refreshes only when dependencies change.

AI + Pipelines

AI accelerates code generation. Pipelines prevent that speed from becoming chaos.

The faster AI generates code, the stricter your pipeline must become. AI doesn't understand your entire system; your pipeline does.

AI-Specific Risks

Hidden Bugs

AI generates code that compiles and passes its own tests. The tests only cover happy paths. Edge cases, null inputs, network failures, concurrent access: these go untested.

Example: AI writes a unit test for addUser() that only tests the success case. The function crashes on duplicate email, network timeout, and invalid input; none of these are caught.

Security Vulnerabilities

Studies show 45% of AI-generated code contains OWASP Top 10 vulnerabilities. AI replicates patterns from training data, including insecure patterns.

typescript 
// AI generates this (SQL injection vulnerability)
const user = await db.query(
  `SELECT * FROM users WHERE id = ${req.params.id}`
);

// Should be (parameterized query)
const user = await db.query(
  'SELECT * FROM users WHERE id = $1',
  [req.params.id]
);

Duplicated Logic

AI looks for prior art and replicates patterns without checking if the same utility already exists. You end up with 4 copies of formatDate() across different files. Fix the bug in 1 place, miss it in 3 others.

Breaking Shared Utilities

AI modifies a shared utility without realizing 8 other parts of the system depend on it. Adding .trim() to a shared JSON parser seems safe, until it breaks 6 other services that relied on the original behavior.

The Technical Debt Timeline

Months 1–3: The Honeymoon

Features ship 5x faster. Tests pass. Stakeholders are thrilled. Cost overhead: ~12%

Months 4–8: Debt Accumulates

Small bugs appear in 3+ places (duplicated logic). Security patches pile up. Developers slow down: "Why does changing this break that?" Cost overhead: ~25%

Months 9–18: Debt Crisis

Major features take 4x longer because the codebase is tangled. On-call rotations overloaded. New developers take weeks to understand the codebase. Total cost: 4x baseline.

The AI PR Review Checklist

  1. 1
    Check CI first

    Does it compile? Do all existing tests pass? Red CI = send back immediately.

  2. 2
    Hunt for duplicated code

    Search for functions the AI might have reimplemented. (grep -rn "function formatDate")

  3. 3
    Inspect security patterns

    Parameterized queries? Input validation? No hardcoded secrets?

  4. 4
    Examine error handling

    Try/catch blocks? What happens in the catch? No silent catch {}.

  5. 5
    Verify API calls

    AI invents methods that don't exist. Check actual library docs for your version.

  6. 6
    Match to requirements

    Does the code do what was actually asked? Did AI go off-spec?

  7. 7
    Check test coverage

    Did AI write tests for the new code? Do the tests cover edge cases?

Never Let AI Commit Directly to Main

An AI agent that can commit directly to main is a production incident waiting to happen. All AI-generated code must:

  • Create a feature branch
  • Open a PR
  • Pass all CI checks
  • Get human code review
  • Only then merge

With branch protection rules and required reviews, even an autonomous AI agent cannot force-merge. It must convince CI and humans first.

Test coverage isn't just about quality: it's a safety mechanism for AI-assisted development.
High test coverage teaches AI what "correct" looks like. When the test suite enforces error handling, null checks, and edge cases, AI-generated code that mimics existing patterns inherits those safety properties. Set a coverage gate of 80%+ in your branch protection rules.

AI increases coding speed. Pipelines protect software quality. The two are partners, not adversaries.

Hands-On

Build Your Pipeline

Configuration is where most projects quietly stall. Both blueprints below pair exact terminal commands with copy-ready AI prompts for every step that benefits from one. Hand the prompts to Claude Code, Cursor, or Gemini and let your agent generate the boilerplate while you stay in control of the decisions.

Scenario 1

Starting Fresh — The Greenfield Blueprint

Run each step in sequence. Terminal blocks go directly into your project folder. Prompt blocks are copy-ready for your AI coding agent: paste them into Claude Code, Cursor, or Gemini and let it generate the config files while you review and move on.

  1. 1
    Secure the Cloud & Local Plumbing

    Go to GitHub.com and create a completely blank repository. Do not check the box for a README, license, or .gitignore. Copy the repository URL. Then execute this structural block in your terminal:

    bash 
    # 1. Create directory and initialize Git
mkdir my_new_app && cd my_new_app
git init

# 2. Wire the remote cloud connection
git remote add origin https://github.com/your-username/my_new_app.git

# 3. Create the integration baseline with an empty commit
git checkout -b dev
git commit --allow-empty -m "chore: initial repository infrastructure blueprint"

# 4. Sprout the environmental pipeline branches
git branch qa
git branch staging
git branch main

# 5. Push all matching local branches to your cloud origin at once
git push -u origin --all
  2. 2
    Establish the Compilation and Style Standard

    Open your AI coding agent (Claude Code, Cursor, or Gemini) directly at the root of your fresh directory. Ensure you are on the dev branch (git checkout dev).

    Prompt to AI Agent

    We are starting a brand-new software project built on TypeScript, Node.js, and Vite/React. We are currently working on the local 'dev' branch.

    Your task is to establish our codebase architecture standards. Please generate:

    1. A strict package.json with scripts: "build": "vite build", "lint": "eslint . --ext .ts,.tsx", "typecheck": "tsc --noEmit".
    2. A strict, production-ready tsconfig.json with noImplicitAny and strictNullChecks active.
    3. Production-ready configurations for ESLint and Prettier.

    Do not write any application feature code or user interfaces yet. Only install dependencies and write configuration infrastructure files.

  3. 3
    Add the Automated Safety Net

    Before a single file can move, we must give the pipeline engine something to verify. Without tests, the CI engine will crash on empty commands.

    Prompt to AI Agent

    Configure our testing infrastructure using Vitest.

    1. Install vitest as a development dependency.
    2. Add a script to package.json: "test": "vitest run".
    3. Create a core folder at the root named src/ and a utility subfolder src/utils/.
    4. Inside src/utils/, write a dummy utility file math.ts with a simple add function, and a companion test file math.test.ts asserting that 2 + 2 equals 4.

    Verify that your scripts are syntactically sound.

  4. 4
    Forge the Cloud Pipeline Automation

    Now we write the exact instructions that GitHub will execute inside its virtual cloud machines whenever code moves.

    Prompt to AI Agent

    Create a completely automated GitHub Actions pipeline configuration file at .github/workflows/ci.yml.

    1. Trigger Name: "CI/CD Pipeline Engine"
    2. Triggers: Any push to dev, and any Pull Request opened against dev, qa, staging, or main.
    3. Environment: ubuntu-latest with Node.js version 20.
    4. Performance: Implement npm caching (cache: npm) for fast execution speeds.
    5. Steps: Checkout code → Setup Node.js → Install via npm ci → npm run lint → npm run typecheck → npm test → npm run build.

    Ensure absolute compliance with YAML layout rules — two spaces for indentation, zero tab characters.

    Now push this configuration to fire the initial test flight:

    bash 
    git add .
git commit -m "chore: activate automated github actions ci pipeline engine"
git push origin dev
  5. 5
    Enforce Repository Rulesets

    Go to GitHub.com → Your Repository → Settings → Rules → Rulesets. Click New ruleset → New branch ruleset.

    Ruleset Name: Core Guardrails  ·  Enforcement: Active  ·  Target branches: Include by pattern — main, staging, qa, dev
    • Require a pull request before merging (forces code isolation)
    • Require status checks to pass before merging — click Add checks and search for your exact job name from the YAML file. This keeps the merge button locked if a check fails.
    • Block force pushes (prevents erasing history)
Scenario 2

Retrofitting a Massive Legacy Project — The Brownfield Blueprint

Already live, zero pipeline, zero tests. Injecting everything at once risks a production outage. The strategy: use AI prompts to build the pipeline infrastructure on a parallel branch and slide it underneath the running app before anyone notices. Same pattern as above: terminal blocks for you, prompt blocks for your agent.

text 
 [Legacy Project: Only main branch]
               │
               ▼
 [1. Pull Dev Branch] ──► [2. Install Strict Linters & Build Checks]
                                                │
                                                ▼
 [4. Merge dev → main via PR] ◄── [3. Inject ci.yml Automation Engine]
  1. 1
    Create the Containment Zone

    Stop writing code on your live main branch immediately. Create an isolation layer to safely build the pipeline.

    bash 
    # Stand safely on main and ensure you have the latest code
git checkout main
git pull origin main

# Create the dev environment branch directly from main
git checkout -b dev
git push -u origin dev

# Sprout qa and staging from the exact same point

git push origin qa
git push origin staging
  2. 2
    Establish Legacy Quality Baselines

    Running a strict linter on a massive existing codebase for the first time will likely throw 500+ formatting errors. Teach the AI to set up checking tools gently so they do not block immediate builds.

    Make sure you are on your local dev branch (git checkout dev).

    Prompt to AI Agent

    We are retrofitting an existing, massive production application with a CI pipeline. We are currently on the dev branch.

    1. Open package.json. Add standard scripts for "build", "typecheck": "tsc --noEmit", and "lint". If lint rules do not exist, create a baseline ESLint profile configured as 'warn' rather than 'error' so legacy style choices do not break immediate deployments.
    2. Run a dry compilation check using terminal tool execution (npm run build and npm run typecheck).
    3. If any compilation or type errors exist in our legacy code, list the critical files but do not alter them yet.

    Our goal is simply to ensure that our current production state can successfully execute a standard compile script.

  3. 3
    Inject the Automation Framework

    Since this project has no test folder yet, explicitly command the YAML pipeline to bypass tests — preventing false pipeline failures.

    Prompt to AI Agent

    Create a custom GitHub Actions file at .github/workflows/ci.yml tailored for this legacy repository codebase.

    1. Triggers: Active on pushes to dev, or Pull Requests targeted at dev, qa, staging, or main.
    2. Steps: Checkout repository, configure Node.js 20 with npm caching enabled, run npm ci.
    3. Validations: Execute npm run lint, npm run typecheck, and npm run build.
    4. CRITICAL: Do NOT include an npm test step — this legacy project does not possess a tests folder yet. We will introduce tests incrementally.

    Ensure the YAML script structure is completely valid.

  4. 4
    The Core Merge — Activating the Protection

    Push your newly created pipeline files up to your cloud repository:

    bash 
    git add .
git commit -m "infrastructure: retrofit automated git branches and ci core validations"
git push origin dev

    Now complete the containment loop:

    • Open GitHub.com and create a Pull Request: dev → main
    • Watch GitHub Actions run on your massive codebase for the first time — it will scan your types, verify lint warnings, and run a production compile
    • Once it turns green, click Merge
  5. 5
    Activate the Lock and Begin Feature Branches

    Now that the pipeline configuration exists on your primary live branch, activate GitHub Rulesets (exactly like Step 5 in Scenario 1) to protect main, staging, qa, and dev.

    From this exact moment forward, writing code directly on the production branch is completely locked down. Your massive legacy app is fully protected inside an enterprise-grade automated pipeline. To build any feature going forward:

    bash 
    git checkout dev
git pull origin dev
git checkout -b feature/my_new_app-squads

After Setup: Separating Your Environments

Once either blueprint above is running, you face the next critical infrastructure challenge: keeping Staging from ever touching Production data. Apply the following three-pillar system regardless of which scenario you followed — it handles the environment boundary cleanly using Supabase for database isolation, Railway for environment-scoped hosting, and GitHub Actions for automated promotion.

The Rule of Complete Separation

Staging must never touch Production data. A staging bug that mutates a production database is not a staging problem: it is a production incident. The three pillars below enforce this boundary at the infrastructure level, not by convention.

Pillar 1

Separating Your Databases (Supabase)

Before your first line of feature code, decide which database situation you are in. Pick the scenario that matches and follow it exactly — both paths end at the same destination: schema in code, two isolated environments, and Claude writing every migration.

Scenario 1

Starting a Brand-New Project (Two Fresh Databases)

  1. 1
    Set up the spaces

    Create two separate, completely blank projects in your Supabase dashboard — one for staging, one for production. Leave them entirely empty. Do not create any tables yet.

    • my_app-staging — blank Supabase project
    • my_app-prod — blank Supabase project
  2. 2
    The Master Initialization Prompt for Claude

    Open a fresh chat with Claude Code or Cursor at your project root. Paste this prompt exactly — it forces Claude to write your migration rules into CLAUDE.md with a structured format, initialize the Supabase project locally, and lock the baseline into version history before a single feature gets built.

    Prompt to AI Agent

    Create a CLAUDE.md file at the project root. Add a DATABASE MIGRATION RULES section with these exact rules — no exceptions:

    • ALL schema changes MUST use: npx supabase migration new <descriptive_name>
    • Migration names must describe the change — e.g. create_users_table, add_stripe_id_to_profiles — never generic names like schema_update or fix
    • NEVER edit the Supabase dashboard directly — all schema lives in code
    • NEVER edit an existing migration file — always create a new incremental one
    • Every migration must be committed in the same Git commit as the code that depends on it

    After writing CLAUDE.md, run npx supabase init, then npx supabase migration new baseline_schema. Leave the baseline file empty — it anchors our version history. Commit both files together.

Scenario 2

Migrating an Existing Project (One Active DB ──► Adding a Twin)

  1. 1
    Set up the spaces

    Your current active database becomes Staging. Spin up a brand-new, completely empty Supabase project for Production. Then wire your staging connection string into GitHub secrets so the CLI can reach it.

    • Existing database → now your Staging environment
    • New blank Supabase project → your Production environment
    • Add STAGING_DATABASE_URL to GitHub → Settings → Secrets and Variables → Actions → Repository Secrets
  2. 2
    The Master Transition Prompt for Claude

    Open your chat with Claude Code or Cursor. This prompt first updates CLAUDE.md with the multi-environment rules, then pulls your entire existing live schema into a baseline migration file — so every future migration starts from a documented, version-controlled state.

    Prompt to AI Agent

    Update our CLAUDE.md file — create it if it doesn't exist. Add a DATABASE MIGRATION RULES section with these exact rules — no exceptions:

    • ALL schema changes MUST use: npx supabase migration new <descriptive_name>
    • Migration names must describe the actual change — e.g. create_users_table, add_avatar_url_to_profiles — never generic names
    • NEVER edit the Supabase dashboard directly — all schema lives in code
    • NEVER edit an existing migration file — only create new incremental ones
    • Every migration must be committed alongside the app code that depends on it
    • To apply migrations to staging locally: npx supabase db push --db-url $STAGING_DATABASE_URL

    After updating CLAUDE.md, snapshot our existing staging schema into version control:

    npx supabase db pull --db-url $STAGING_DATABASE_URL

    This writes our entire current schema into supabase/migrations/ as our locked baseline. Commit CLAUDE.md and this baseline migration together immediately.

The 4 Secrets to Add to GitHub

While Claude handles writing the rules and generating the SQL files, drop these four keys into your GitHub repository under Settings → Secrets and Variables → Actions → Repository Secrets:

Secret Name What to Copy & Paste Inside
STAGING_DATABASE_URL Connection string from your Staging Supabase project (Settings → Database). Include your password.
PROD_DATABASE_URL Connection string from your Production Supabase project (Settings → Database). Include your password.
RAILWAY_STAGING_TOKEN Scoped project token generated inside your Railway Staging environment settings.
RAILWAY_PROD_TOKEN Scoped project token generated inside your Railway Production environment settings.

The Cloud Pipeline Takes Care of the Rest

The moment you push, the CI/CD pipeline reads the exact migration files Claude created and applies them in order:

text 
               ┌──► Merged into 'staging' ──► Auto Syncs Staging DB  ──► Deploys Staging App
 [Claude SQL] ─┤
               └──► Merged into 'main'    ──► Auto Syncs Production DB ──► Deploys Production App
  • Merge to Staging: Pipeline reads Claude's migration file and injects the changes into the Staging database instantly.
  • Merge to Production (Main): That exact same file runs against your Production database before any updated code goes live on Railway.
Pillar 2

Multi-Environment Hosting (Railway)

Railway lets you run multiple environments inside a single project. This keeps your architecture visible in one dashboard view rather than scattered across separate projects.

  1. 1
    Structure the Railway Environments
    1. Open your Railway dashboard and create a single project named my_new_app.
    2. Inside that project, you will see a default environment named production.
    3. Click the environment dropdown at the top-left, select New Environment, and name it staging.
  2. 2
    Connect Your Git Branch to the Staging Environment
    1. Look at the top left of your Railway canvas and make sure you have switched the dropdown from production to your newly created staging environment.
    2. Click directly on your application's service block on the canvas to open its side panel.
    3. In that side panel, look at the tabs across the top (usually Deployments, Variables, Metrics, Settings). Click on Settings.
    4. Scroll down slightly until you see the Source section.
    5. You will see your connected GitHub repository name, and directly below it a Branch setting — it defaults to copying whatever your production instance was using, like main or master.
    6. Click that dropdown, type or select staging, and save it.

    Inject the Environment Secrets

    In each Railway environment's Variables tab, add the corresponding Supabase credentials you gathered in Pillar 1:

    NEXT_PUBLIC_SUPABASE_ANON_KEY = Staging Key NEXT_PUBLIC_SUPABASE_ANON_KEY = Prod Key
Pillar 3

The Automation Pipeline

GitHub Actions handles the heavy lifting. It validates every push, then deploys automatically: to Staging when code reaches the staging branch, and to Production when it reaches main.

Before you configure the YAML: go to your Railway Account Settings > Tokens and generate a CLI token. Then add it to your GitHub Repository Settings > Secrets and Variables > Actions as a Repository Secret named RAILWAY_TOKEN.

Update your .github/workflows/ci.yml with this environment-aware pipeline:

yaml 
name: CI/CD Deployment Pipeline

on:
  push:
    branches: [dev, staging, main]
  pull_request:
    branches: [dev, staging, main]

jobs:
  validate-and-test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: Install Dependencies
        run: npm ci

      - name: Run Linters
        run: npm run lint

      - name: Run Type-Checking
        run: npm run typecheck

  deploy-to-staging:
    needs: validate-and-test
    if: github.ref == 'refs/heads/staging' && github.event_name == 'push'
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      # 1. AUTOMATIC DATABASE SYNC: Push migrations made by Claude to Staging DB
      - name: Sync Staging Database Schema
        run: npx supabase db push --db-url "${{ secrets.STAGING_DATABASE_URL }}"

      # 2. APP DEPLOYMENT: Ship frontend code to Railway Staging
      - name: Install Railway CLI
        run: npm install -g @railway/cli

      - name: Deploy to Railway Staging
        run: railway deploy --environment staging
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}

  deploy-to-production:
    needs: validate-and-test
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      # 1. AUTOMATIC DATABASE SYNC: Push migrations made by Claude to Production DB
      - name: Sync Production Database Schema
        run: npx supabase db push --db-url "${{ secrets.PROD_DATABASE_URL }}"

      # 2. APP DEPLOYMENT: Ship frontend code to Railway Production
      - name: Install Railway CLI
        run: npm install -g @railway/cli

      - name: Deploy to Railway Production
        run: railway deploy --environment production
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}

The Day-to-Day Workflow

Once your architecture is locked in, the cycle becomes a smooth, automated loop. You stop worrying about breaking the database and focus entirely on features. Here is exactly how a new idea goes from your head to production.

  1. 1
    The Starting Line: Branching & Context

    Every new task begins with a clean slate. You never work directly on dev — you work in a bubble.

    bash 
    git checkout -b feature/user-streaks
    Prompt to AI Agent

    Check CLAUDE.md for our current rules. We are starting the 'User Streaks' feature. Plan the steps and let's get started.

    Claude reads CLAUDE.md, sees the Database Migration Rules, and knows the first step must be a migration file — not a dashboard click.

  2. 2
    The Build: AI-Driven Migrations

    Claude builds the feature. Because the rules are in CLAUDE.md, it handles the CLI commands automatically.

    • Migration: Claude runs npx supabase migration new add_streaks_table and writes the SQL inside that file.
    • Code: Claude builds the frontend and backend logic on your branch.
    • Local sync: Run npx supabase db reset locally to confirm the migration works on your machine before pushing.
  3. 3
    The Auto-Merge: From Branch to dev

    You push your branch. You don't navigate GitHub to merge — automation handles it.

    bash 
    git add .
git commit -m "feat: implemented user streak logic and schema"
git push origin feature/user-streaks

    A GitHub Action triggers on the feature/* push, runs a quick check, and automatically merges the code into dev.

  4. 4
    The Staging Deployment: Automated Testing

    With code in dev, the CI pipeline takes over and promotes it to Staging.

    • CI trigger: The push to dev fires ci.yml.
    • Database migration: The pipeline grabs STAGING_DATABASE_URL and runs supabase db push — your Staging Supabase gets the new Streaks table instantly.
    • App deployment: RAILWAY_STAGING_TOKEN pushes the new code to your Railway Staging environment.
    • Result: Your team opens the Staging URL and tests the live feature.
  5. 5
    The Final Step: Manual Promotion to Production

    Production never updates on a dev push. You promote only when the team gives the thumbs up.

    • Open a Pull Request from dev ──► main on GitHub.
    • Click Merge Pull Request.
    • ci.yml detects the push to main, grabs PROD_DATABASE_URL, and runs the migration files against your Production Supabase.
    • RAILWAY_PROD_TOKEN deploys the final build to your live Railway app.
